1. 포트폴리오 디펜스

제출 산출물(분석서 55p + 테스트 설계서)을 읽은 면접관이 던질 예상 질문과 답변 가이드.

Q1. 이 분석서를 왜 만들었나요?

핵심 메시지: 게임 QA는 게임을 구조적으로 이해해야 한다.

"개발 QA로 합류한다면, 전투 시스템의 구조를 이해하는 게 첫 단계라고 생각했습니다. 분석서는 '이 게임의 전투를 어떻게 테스트할 것인가'에 대한 사전 연구입니다. 실제로 분석서를 쓰면서 DEF 기반 공식이 7캐릭터만 해당하는 예외 경로라는 점, 합연산과 곱연산의 순서가 밸런스에 큰 영향을 준다는 점 등 테스트 우선순위가 자연스럽게 도출되었습니다."

보충 포인트:

  • 단순 플레이가 아닌 시스템 분해 → 테스트 전략 수립 과정을 보여주는 것이 목적
  • 공개 데이터(prydwen.gg, czncompass, 나무위키)만으로도 역산/분석이 가능하다는 것 자체가 QA적 접근
  • 분석서 작성 중 발견한 것들이 테스트 설계서의 리스크 분석으로 직접 연결됨

Q2. 데미지 공식은 어떻게 검증했나요?

핵심 메시지: 한계를 인정하되, 그 한계가 QA의 존재 이유임을 연결.

"prydwen.gg에 공개된 데미지 공식을 기반으로 했습니다. 외부에서 역산한 공식이므로 실제 서버 구현과 차이가 있을 수 있다는 점은 설계서 한계 섹션에 명시했습니다. 하지만 그 '차이를 찾아내는 것' 자체가 QA의 역할이라고 생각합니다. 공개 공식 대비 인게임 결과가 다른 케이스가 있다면, 그것이 버그인지 의도된 숨겨진 보정값인지를 검증하는 게 개발 QA의 일입니다."

공식 요약 (즉시 설명할 수 있어야 함):

경로 공식 해당
ATK 기반 카드배율 × ATK × 0.35 × 크리티컬배수 21캐릭터
DEF 기반 카드배율 × (ATK×0.3 + DEF×2.1) × 0.35 × 크리티컬배수 7캐릭터
  • 크리티컬 배수 = 1 + (CC/100) × ((150+CD)/100 - 1)
  • 합연산(+X%) 먼저 → 곱연산(X% 증가) 나중
  • 복수 곱연산끼리는 합산 후 1회 적용

Q3. 테스트 기법 선택 근거는?

핵심 메시지: 리스크 분석에서 출발 → 등급별 기법 매핑.

"리스크 분석을 먼저 수행하고, 리스크 등급에 따라 테스트 기법을 매핑했습니다."

리스크 대상 기법 이유
Critical DEF 기반 공식 동치 분할 + 경계값 28명 중 7명만 해당하는 소수 경로, 계수 2개 추가
High 시너지 중첩 Pairwise 5×6×4×3=360 조합을 25개로 축소 (93% 감소)
Medium 크리티컬 계산 경계값 분석 CC/CD 외부 주입, 극단값 미검증
Low 다단히트 속성 기반 불변 조건 일괄 검증 (N=1000)

Pairwise 선택 근거: 결함의 대부분이 2개 변수의 상호작용에서 발생한다는 경험적 근거(Kuhn et al., 2004). 3-way로 확장하면 ~100개 케이스가 필요하지만, 3개 변수 동시 상호작용의 Critical 리스크가 식별되지 않아 2-way로 충분하다고 판단.

Q4. DEF 기반 캐릭터의 리스크가 왜 Critical인가요?

요인 설명
소수 경로 28명 중 7명(25%) — 테스트 누락 확률 높음
넓은 결함 표면적 ATK 기반(계수 1개) vs DEF 기반(계수 2개: 0.3, 2.1)
극단적 영향 DEF=0이면 데미지 70% 감소 (ATK 기반 대비)
코드 분기 위험 defBasedDamage 트레이트 판별 실패 시 전원 영향
예외적 매핑 나르쟈·티페라는 컨트롤러인데 DEF 기반 — 클래스≠공식

"클래스와 데미지 공식이 1:1 매핑이 아닌 예외가 있어서, 클래스만 보고 테스트 커버리지를 판단하면 놓칠 수 있습니다."

Q5. 가장 의미 있는 발견은?

상황에 따라 택 1.

발견 A — 합연산 vs 곱연산 순서 의존성

"합연산(+X%)을 먼저, 곱연산(X% 증가)을 나중에 적용하는데, 이 순서가 바뀌면 최종 데미지가 달라집니다. 시너지 2개 이상 중첩 시 반올림 차이로 ±1 오차가 발생할 수 있고, 이게 유저 리포트의 원인이 될 수 있습니다."

발견 B — 바리에이션 52%가 역할 전환

"560개 번뜩임 바리에이션 중 52%가 캐릭터의 역할을 바꿉니다. 원본 카드만 테스트하면 안 되고, 바리에이션별로 별도 검증이 필요합니다. 유형 전환 82건, 유틸리티 전용 207건, 타입 변경 중 Skill→Upgrade 34건이 최다였습니다."

발견 C — CC/CD 외부 주입 문제

"크리티컬이 캐릭터 고유값이 아니라 Potential 노드에서 주입됩니다. 현재 모든 캐릭터가 동일한 CC=10/CD=12를 가지고 있어서, 이것이 진짜 기본값인지 Potential 미설정 상태인지 구분이 안 됩니다."

Q9. 밸런스에서 문제라고 생각하는 부분?

주의: "밸런스가 나쁘다"가 아닌 **"QA 관점에서 검증이 필요한 영역"**으로 프레이밍.

답변 방향:

  • "피해 배율 상위 5명 평균 1216% vs 하위 5명 298%로 4.1배 격차가 있습니다. 의도된 역할 분화인지, 추가 보정이 필요한지는 내부 설계 의도를 알아야 판단할 수 있습니다."
  • "AP 효율 이상치로 허무의 잔상(카일론) 360%/AP, 중화기 전문가(베릴) 350%/AP가 있는데, 해당 캐릭터의 핵심 강점으로 의도된 건지 확인이 필요합니다."

절대 "X 캐릭터가 OP다/너프해야 한다"고 말하지 않는다. QA는 판단이 아닌 데이터 제시.

Q10. 최근 업데이트 내용?

면접 전날 확인 필수. 최소 최근 2~3회 패치 내용은 숙지.

3. QA 실무 역량 — STAR 프레임워크

펄어비스 7년 경력에서 끌어올 사례. STAR(상황-과제-행동-결과) 형식으로 정리.

준비할 사례 5가지

S1. 가장 임팩트 있었던 버그 발견

STAR 내용
상황 (프로젝트/시점)
과제 (발견해야 했던 문제)
행동 (구체적으로 한 것)
결과 (성과, 가능하면 숫자)

S2. 개발팀과 의견 충돌

STAR 내용
상황 (어떤 이슈에서 충돌)
과제 ("버그 아님" vs QA 판단)
행동 (설득/에스컬레이션/데이터 제시)
결과 (합의 과정, 최종 결론)

S3. 테스트 프로세스 개선

STAR 내용
상황 (비효율이 있던 기존 프로세스)
과제 (개선 목표)
행동 (체크리스트 도입, 자동화, 도구 등)
결과 (시간 절감, 버그 발견율 변화 등)

S4. 일정 촉박 시 우선순위 결정

STAR 내용
상황 (일정 압박 상황)
과제 (전수 테스트 불가, 선택 필요)
행동 (리스크 기반 우선순위 결정 방법)
결과 (놓친 것 없이 출시, 또는 교훈)

S5. 기술적으로 도전적이었던 테스트

STAR 내용
상황 (복잡한 시스템)
과제 (일반적 방법으로 테스트 불가)
행동 (창의적/기술적 접근)
결과 (발견한 결함, 개선된 프로세스)

펄어비스 NDA 주의: 민감 정보(미출시 프로젝트명, 내부 수치 등)는 일반화하여 답변. "A 프로젝트에서" 또는 "대규모 MMORPG에서"로 대체 가능.

질문 답변 방향

# 질문 의도
1 "현재 QA팀의 규모와 개발팀과의 협업 방식은?" 팀 구조 파악, 내 배치 위치
2 "개발 QA와 일반 QA의 역할 구분이 어떻게 되어 있나요?" 포지션 기대치 확인
3 "자동화 테스트나 CI/CD에 QA가 참여하는 부분이 있나요?" 기술적 성장 가능성
4 "밸런스 패치의 의사결정 프로세스에 QA가 어떻게 기여하나요?" QA의 실질적 영향력
5 "현재 가장 QA 리소스가 필요한 영역은 어디인가요?" 바로 기여할 수 있는 곳

핵심 마인드셋 3가지

면접 전체를 관통하는 메시지:

1. 구조적 분해 능력 "나는 게임을 단순히 즐기는 게 아니라, 시스템을 분해하고 테스트 전략을 수립할 수 있다."

> 셋째, **유저 관점 피드백** — 기능은 정상이지만 유저 경험이 좋지 않은 부분을 짚어내는 것.

### 1.2 "히스토리 중심 이슈 트래킹이란?"

> "단순히 '버그 발견 → 등록 → 수정 확인'이 아니라, **그 버그가 왜 발생했고, 같은 원인의 다른 버그가 없는지, 수정 후 재발하지 않는지**까지 추적하는 것이라고 이해합니다.
>
> 실제로 CZN에서 레노아의 즉결 심판 번뜩임 중첩 시 피해량 미증가 버그가 있었고, 진혼의 탄환 약점 피해량 미적용 이슈도 있었습니다. 이런 케이스는 '레노아만의 문제'가 아니라 **카드 효과 중첩 로직 전체**에 같은 패턴이 있는지 확인해야 합니다. 히스토리를 추적하면 '이 버그는 3월 패치에서 수정됐는데, 4월에 비슷한 구조의 다른 카드에서 재발했다'는 식으로 패턴을 잡을 수 있습니다.
>
> 펄어비스에서도 이슈 트래킹 시 관련 이슈를 링크하고, 수정 빌드에서 회귀 테스트를 수행하는 방식으로 히스토리를 관리했습니다."

## 2. 포트폴리오 디펜스

제출 산출물(분석서 55p + 테스트 설계서)을 읽은 면접관이 던질 예상 질문과 답변 가이드.

### Q1. 이 분석서를 왜 만들었나요?

**핵심 메시지**: 게임 QA는 게임을 구조적으로 이해해야 한다.

> "개발 QA로 합류한다면, 전투 시스템의 구조를 이해하는 게 첫 단계라고 생각했습니다. 분석서는 '이 게임의 전투를 어떻게 테스트할 것인가'에 대한 사전 연구입니다. 실제로 분석서를 쓰면서 DEF 기반 공식이 7캐릭터만 해당하는 예외 경로라는 점, 합연산과 곱연산의 순서가 밸런스에 큰 영향을 준다는 점 등 테스트 우선순위가 자연스럽게 도출되었습니다."

**보충 포인트:**
- 단순 플레이가 아닌 **시스템 분해 → 테스트 전략 수립** 과정을 보여주는 것이 목적
- 공개 데이터(prydwen.gg, czncompass, 나무위키)만으로도 역산/분석이 가능하다는 것 자체가 QA적 접근
- 분석서 작성 중 발견한 것들이 테스트 설계서의 리스크 분석으로 직접 연결됨

---

### Q2. 데미지 공식은 어떻게 검증했나요?

**핵심 메시지**: 한계를 인정하되, 그 한계가 QA의 존재 이유임을 연결.

> "prydwen.gg에 공개된 데미지 공식을 기반으로 했습니다. 외부에서 역산한 공식이므로 실제 서버 구현과 차이가 있을 수 있다는 점은 설계서 한계 섹션에 명시했습니다. 하지만 그 '차이를 찾아내는 것' 자체가 QA의 역할이라고 생각합니다. 공개 공식 대비 인게임 결과가 다른 케이스가 있다면, 그것이 버그인지 의도된 숨겨진 보정값인지를 검증하는 게 개발 QA의 일입니다."

**공식 요약** (즉시 설명할 수 있어야 함):

| 경로 | 공식 | 해당 |
|------|------|------|
| ATK 기반 | `카드배율 × ATK × 0.35 × 크리티컬배수` | 21캐릭터 |
| DEF 기반 | `카드배율 × (ATK×0.3 + DEF×2.1) × 0.35 × 크리티컬배수` | 7캐릭터 |

- 크리티컬 배수 = `1 + (CC/100) × ((150+CD)/100 - 1)`
- 합연산(+X%) 먼저 → 곱연산(X% 증가) 나중
- 복수 곱연산끼리는 합산 후 1회 적용

### Q3. 테스트 기법 선택 근거는?

**핵심 메시지**: 리스크 분석에서 출발 → 등급별 기법 매핑.

> "리스크 분석을 먼저 수행하고, 리스크 등급에 따라 테스트 기법을 매핑했습니다."

| 리스크 | 대상 | 기법 | 이유 |
|--------|------|------|------|
| **Critical** | DEF 기반 공식 | 동치 분할 + 경계값 | 28명 중 7명만 해당하는 소수 경로, 계수 2개 추가 |
| **High** | 시너지 중첩 | Pairwise | 5×6×4×3=360 조합을 25개로 축소 (93% 감소) |
| **Medium** | 크리티컬 계산 | 경계값 분석 | CC/CD 외부 주입, 극단값 미검증 |
| **Low** | 다단히트 | 속성 기반 | 불변 조건 일괄 검증 (N=1000) |

**Pairwise 선택 근거**: 결함의 대부분이 2개 변수의 상호작용에서 발생한다는 경험적 근거(Kuhn et al., 2004). 3-way로 확장하면 ~100개 케이스가 필요하지만, 3개 변수 동시 상호작용의 Critical 리스크가 식별되지 않아 2-way로 충분하다고 판단.

### Q4. DEF 기반 캐릭터의 리스크가 왜 Critical인가요?

| 요인 | 설명 |
|------|------|
| 소수 경로 | 28명 중 7명(25%) — 테스트 누락 확률 높음 |
| 넓은 결함 표면적 | ATK 기반(계수 1개) vs DEF 기반(계수 2개: 0.3, 2.1) |
| 극단적 영향 | DEF=0이면 데미지 **70% 감소** (ATK 기반 대비) |
| 코드 분기 위험 | `defBasedDamage` 트레이트 판별 실패 시 전원 영향 |
| 예외적 매핑 | 나르쟈·티페라는 컨트롤러인데 DEF 기반 — 클래스≠공식 |

> "클래스와 데미지 공식이 1:1 매핑이 아닌 예외가 있어서, 클래스만 보고 테스트 커버리지를 판단하면 놓칠 수 있습니다."

### Q5. 가장 의미 있는 발견은?

상황에 따라 택 1.

**발견 A — 합연산 vs 곱연산 순서 의존성**

> "합연산(+X%)을 먼저, 곱연산(X% 증가)을 나중에 적용하는데, 이 순서가 바뀌면 최종 데미지가 달라집니다. 시너지 2개 이상 중첩 시 반올림 차이로 ±1 오차가 발생할 수 있고, 이게 유저 리포트의 원인이 될 수 있습니다."

**발견 B — 바리에이션 52%가 역할 전환**

> "560개 번뜩임 바리에이션 중 52%가 캐릭터의 역할을 바꿉니다. 원본 카드만 테스트하면 안 되고, 바리에이션별로 별도 검증이 필요합니다. 유형 전환 82건, 유틸리티 전용 207건, 타입 변경 중 Skill→Upgrade 34건이 최다였습니다."

**발견 C — CC/CD 외부 주입 문제**

> "크리티컬이 캐릭터 고유값이 아니라 Potential 노드에서 주입됩니다. 현재 모든 캐릭터가 동일한 CC=10/CD=12를 가지고 있어서, 이것이 진짜 기본값인지 Potential 미설정 상태인지 구분이 안 됩니다."

| **스트레스/붕괴** | HP와 별개의 패배 조건 — 비선형 위협 관리 |
| **행동 카운트** | 몬스터가 턴이 아닌 행동 카운트로 움직여 무한 루프 방지 |

### 3.3 현재 게임에서 가장 불편한 점은? (유저 관점 피드백)

> **주의**: "문제다"가 아닌 **"QA 관점에서 검증이 필요한 영역"**으로 프레이밍.

#### 구조적 분석: 이중 장벽

> "CZN은 덱빌딩 로그라이크와 RPG 육성을 결합한 구조인데, 이 결합이 **이중 장벽**을 만든다고 느꼈습니다.
>
> 일반적인 덱빌딩 로그라이크는 세션 안에서 덱을 빌딩하고 클리어하면 끝입니다. 그런데 CZN은 '세이브 데이터'라는 영속적인 결과물이 있고, 이걸 만들려면 카오스 던전을 클리어해야 합니다. 문제는 **카오스 던전 자체가 캐릭터 레벨과 기억의 조각 같은 RPG 육성에 의존한다**는 점입니다. 기본 스탯이 부족하면 높은 등급의 세이브 데이터를 얻을 수 없습니다.
>
> 반대 방향도 있습니다. RPG 육성은 충분한데 세이브 데이터가 부실하면, 대균열 같은 시즌 PvE 콘텐츠에서 고득점을 낼 수 없습니다.
>
> 결국 **콘텐츠 하나를 제대로 플레이하려면 RPG 육성과 세이브 데이터 빌딩 두 시스템을 모두 충족해야 하는 구조**입니다."

#### QA 관점 연결

> "솔직히 이 구조적 문제의 해결방안은 제가 판단하기 어렵습니다. 두 시스템의 결합 자체가 CZN의 정체성이니까요.
>
> 다만 **QA 관점에서는 두 가지를 검증할 수 있다**고 생각합니다. 첫째, **두 시스템의 성장 곡선이 동기화되어 있는지** — 레벨은 올랐는데 세이브 데이터 품질이 못 따라가는 구간, 또는 그 반대 구간이 어디인지를 데이터로 식별하는 것. 둘째, **신규 유저가 첫 번째 유의미한 콘텐츠에 도달하기까지의 경험이 설계 의도와 일치하는지** 검증하는 것입니다.
>
> 해결은 기획의 영역이지만, **문제가 발생하는 구간을 정확히 짚어내는 건 QA의 역할**이라고 생각합니다."

## 4. QA 실무 역량

### 4.1 ISTQB 실무 적용

**질문**: "ISTQB를 취득하면서 배운 점이나 테스트 방법론을 실제 업무에서 적용해 본 적 있나요?"

#### 핵심 메시지

> "ISTQB를 공부하면서 가장 크게 바뀐 건, **테스트를 설계할 때 '왜 이 기법을 쓰는가'에 대한 근거를 갖게 된 것**입니다."

#### 사례 1 — 경계값 분석: 전투력 단계별 버프

> "검은사막 모바일에서 가문 전투력을 합산해서 특정 구간을 넘으면 버프가 부여되는 시스템이 있었습니다. 이 시스템을 테스트할 때 **경계값 분석**을 적용했습니다.
>
> 예를 들어 전투력 구간 경계가 5000이면, 4999/5000/5001에서 버프가 정확히 전환되는지 확인하는 방식입니다. 모든 전투력 수치를 다 확인할 수는 없지만, ISTQB에서 배운 대로 **결함은 구간 경계에서 가장 많이 발생한다**는 원칙에 따라 경계값 위주로 테스트했고, 실제로 경계값에서 버프가 정상 적용되지 않는 케이스를 여러 차례 발견할 수 있었습니다."

#### 사례 2 — 동치 분할: 장비 강화 시스템

> "장비 강화 시스템이 추가될 때, 강화 데이터가 각 장비별로 개별 존재했기 때문에 원칙적으로는 전수 테스트가 필요했습니다. 하지만 장비 종류가 너무 많아서 시간상 불가능했습니다.
>
> 이때 ISTQB의 **동치 분할** 개념을 적용했습니다. 모든 장비를 랜덤으로 뽑는 대신, **등급별 · 부위별로 대표 장비를 3~4개씩 선택**해서 테스트했습니다. 같은 등급 · 같은 부위의 장비는 동일한 강화 로직을 공유할 것이므로, 대표값 검증으로 해당 그룹 전체를 커버할 수 있다는 판단이었습니다.
>
> 단순히 '랜덤으로 골라서 해보자'가 아니라 **'왜 이 장비를 선택했는가'에 근거가 있는 테스트**를 할 수 있었던 게 ISTQB를 배운 효과라고 생각합니다."

#### 마무리 한 문장

> "정리하면, ISTQB에서 배운 가장 중요한 점은 **개별 기법이 아니라 '한정된 시간에 어떤 테스트를 왜 선택하는가'라는 사고방식**이었고, 이걸 실무에서 전투력 경계값 검증이나 장비 강화 대표값 선택 같은 상황에 직접 적용했습니다."

#### ISTQB 꼬리 질문 대비

| 꼬리 질문 | 답변 방향 |
|----------|----------|
| "동치 분할로 대표값만 했는데, 놓친 건 없었나요?" | "동치 분할의 전제는 같은 그룹 내 동일 로직 공유입니다. 리스크가 높은 신규 등급이나 특수 장비는 별도로 추가 확인했습니다" |
| "경계값 분석 외에 다른 기법은?" | 동치 분할 사례 + "리스크가 높은 영역에 더 많은 테스트를 배정하는 리스크 기반 접근도 자연스럽게 적용했습니다" |
| "ISTQB FL만 있나요? AL은?" | 솔직하게 현재 수준 답변. 필요하면 "AL 취득도 계획하고 있다" |

### 4.2 명세서 없는 환경에서의 TC/CL 대응

**질문**: "명세서 기반 TC/CL 제작 경험이 있나요?"

#### 두 프로젝트의 완전히 다른 프로세스

> "솔직히 말씀드리면, 이전 회사에서는 체계적인 명세서가 주어지는 환경이 아니었습니다. 그리고 같은 회사 안에서도 프로젝트마다 QA 프로세스가 완전히 달랐습니다."

**검은사막 모바일:**

> "개발 회의에 참여해서 콘텐츠의 골자를 파악할 수 있었고, 작업 티켓에 작업자가 테스트 필요 항목을 상세하게 기재해야 했습니다. 이게 사실상 명세서 역할을 했고, 이걸 기준으로 테스트했습니다. 부족한 부분은 프로토타입이 나오면 개발자, 기획자와 빠르게 소통하면서 직접 플레이해서 채워넣었습니다."

**붉은사막:**

> "상황이 완전히 달랐습니다. 탑다운 방식의 조직이라 QA는 개발 회의에 참여하지 않았고, BTS 도구도 없어서 작업 티켓 자체가 없었습니다. 대신 슬랙에 4시간 단위로 각 작업자의 작업 현황이 전체 공개되었고, 개발이 완료되면 QA 요청 슬랙방을 통해 테스트를 요청받았습니다.
>
> 다만 요청이 굉장히 광범위한 경우가 많았습니다. 예를 들어 '발사체 동작을 수정했으므로 전반적인 테스트가 필요함' 정도만 주어지는 식입니다. **여기서 어떤 케이스를 어떤 순서로 테스트할지는 온전히 QA의 판단**이었습니다.
>
> 발사체 테스트라면, 발사체가 연관된 AI 스킬, 액션 패턴, 물리 충돌, 기믹 트리거를 먼저 파악하고, 변경된 부분이 영향을 줄 수 있는 범위를 판단해서 우선순위를 정했습니다. 이게 사실상 명세서 없이 테스트 범위를 스스로 설계하는 과정이었습니다."

**마무리:**

> "두 프로젝트 모두 명세서가 없는 환경이었지만, 정보를 얻는 채널과 QA의 역할 범위가 완전히 달랐습니다. 그 경험 덕분에 어떤 프로세스에서든 빠르게 적응할 수 있다고 생각하고, **명세서가 체계적으로 주어지는 환경이라면 더 정확하고 빠르게 TC를 설계할 수 있습니다.** CZN 포트폴리오에서 분석서를 기반으로 테스트 케이스를 도출한 것이 그 증거라고 생각합니다."

#### 명세서 꼬리 질문 대비

| 꼬리 질문 | 답변 방향 |
|----------|----------|
| "TC/CL을 문서로 남긴 경험이 아예 없나요?" | "wiki에 명세가 있는 경우에는 그 기준으로 체크리스트를 만들어 사용한 적이 있습니다. 또한 포트폴리오의 테스트 설계서가 TC를 체계적으로 작성할 수 있다는 증거입니다" |
| "적응 가능한가요?" | "오히려 기대합니다. 명세서가 없는 환경에서도 기준을 세워왔으니, 명세서가 있으면 더 빠르고 정확하게 TC를 설계할 수 있습니다" |
| "두 환경 중 어디가 더 좋았나요?" | "각각 장단점이 있었습니다. 검은사막 모바일은 소통이 자유로워서 유저 관점 피드백을 전달하기 좋았고, 붉은사막은 기능 테스트에 집중할 수 있어서 기술적 판단력이 많이 늘었습니다" |

### 4.3 STAR 사례 프레임워크

펄어비스 7년 경력에서 끌어올 사례. **STAR(상황-과제-행동-결과)** 형식으로 정리.

#### S1. 가장 임팩트 있었던 버그 발견

| STAR | 내용 |
|------|------|
| **상황** | _(프로젝트/시점)_ |
| **과제** | _(발견해야 했던 문제)_ |
| **행동** | _(구체적으로 한 것)_ |
| **결과** | _(성과, 가능하면 숫자)_ |

#### S2. 개발팀과 의견 충돌

| STAR | 내용 |
|------|------|
| **상황** | _(어떤 이슈에서 충돌)_ |
| **과제** | _("버그 아님" vs QA 판단)_ |
| **행동** | _(설득/에스컬레이션/데이터 제시)_ |
| **결과** | _(합의 과정, 최종 결론)_ |

#### S3. 테스트 프로세스 개선

| STAR | 내용 |
|------|------|
| **상황** | _(비효율이 있던 기존 프로세스)_ |
| **과제** | _(개선 목표)_ |
| **행동** | _(체크리스트 도입, 자동화, 도구 등)_ |
| **결과** | _(시간 절감, 버그 발견율 변화 등)_ |

#### S4. 일정 촉박 시 우선순위 결정

| STAR | 내용 |
|------|------|
| **상황** | _(일정 압박 상황)_ |
| **과제** | _(전수 테스트 불가, 선택 필요)_ |
| **행동** | _(리스크 기반 우선순위 결정 방법)_ |
| **결과** | _(놓친 것 없이 출시, 또는 교훈)_ |

| STAR | 내용 |
|------|------|
| **상황** | _(복잡한 시스템)_ |
| **과제** | _(일반적 방법으로 테스트 불가)_ |
| **행동** | _(창의적/기술적 접근)_ |
| **결과** | _(발견한 결함, 개선된 프로세스)_ |

> **펄어비스 NDA 주의**: 민감 정보(미출시 프로젝트명, 내부 수치 등)는 일반화하여 답변. "A 프로젝트에서" 또는 "대규모 MMORPG에서"로 대체 가능.

### "왜 이직하시나요?" 답변 가이드

**하지 말 것:**
- 전 회사 비하, 불만 표현
- "연봉", "워라밸" 같은 조건 중심 답변

**할 것:**
- 새로운 도전 + CZN에 대한 구체적 관심
- "대규모 MMO QA 경험을 바탕으로, 턴제 덱 빌딩이라는 다른 장르에서 QA 역량을 확장하고 싶다"
- "CZN의 전투 시스템이 조합 폭발, 비결정적 빌드 등 QA적으로 깊이 있는 문제를 가지고 있어서 매력적이다"

### 핵심 마인드셋 3가지

면접 전체를 관통하는 메시지:

**1. 구조적 분해 능력**
"나는 게임을 단순히 즐기는 게 아니라, 시스템을 분해하고 테스트 전략을 수립할 수 있다."

**2. 확장 가능성**
"외부에서 이 정도를 했으면, 내부에 합류하면 훨씬 더 할 수 있다."

**3. 즉시 전력**
"7년 실무 경험 + CZN에 대한 깊은 이해 = 합류 즉시 기여 가능."
